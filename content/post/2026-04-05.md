## ACM

### [P2572 [SCOI2010] 序列操作 - 洛谷](https://www.luogu.com.cn/problem/P2572)

- 这题是一个很典型的“维护状态比套模板更重要”的线段树题。
- 操作有区间赋 `0`、区间赋 `1`、区间翻转、查询区间 `1` 的个数、查询区间最长连续 `1` 的长度。
- 如果只维护一个区间和，那显然只能做前四个，最后那个最长连续 `1` 是做不出来的。
- 所以这题的关键就在于：一个节点到底要维护哪些量，才能让儿子的信息正确合并到父亲。

#### 维护的信息

- 你这份代码里维护的是一个二维状态：
  - `sum[0/1]`：区间里 `0/1` 的数量
  - `pre[0/1]`：区间前缀连续 `0/1` 的最长长度
  - `suf[0/1]`：区间后缀连续 `0/1` 的最长长度
  - `mx[0/1]`：区间内连续 `0/1` 的最长长度
- 这个设计其实很自然，因为题目既然有“翻转”操作，那么只维护 `1` 的信息不够，`0` 和 `1` 两套信息最好一起维护。
- 这样区间翻转的时候，不需要重新算，直接把 `0` 和 `1` 的信息整体交换即可。

#### 合并怎么想

- 父亲的 `sum` 显然就是左右儿子对应和相加。
- 而前缀、后缀、最长连续段，和之前最大子段和那题的思路有点像，本质上也是在讨论“跨不跨中点”。
- 以 `mx[1]` 为例，它只可能来自三种情况：
  - 完全在左儿子内部
  - 完全在右儿子内部
  - 横跨中点，也就是左区间后缀连续 `1` 加右区间前缀连续 `1`

\[
mx[1] = \max\bigl(l.mx[1],\ r.mx[1],\ l.suf[1] + r.pre[1]\bigr)
\]

- `mx[0]` 也是一样的。
- `pre` 和 `suf` 的更新也完全是同一个逻辑：如果左儿子整段都是 `1`，那父亲的前缀连续 `1` 就可以延伸到右边去；后缀同理。

#### 为什么翻转很好处理

- 这题最舒服的一点就是翻转操作。
- 因为翻转本质上就是把 `0` 变成 `1`，把 `1` 变成 `0`，所以一个节点被翻转以后：
  - `sum[0]` 和 `sum[1]` 交换
  - `pre[0]` 和 `pre[1]` 交换
  - `suf[0]` 和 `suf[1]` 交换
  - `mx[0]` 和 `mx[1]` 交换
- 所以代码里 `apply_xor` 写得很干净，这题的状态设计也正是妙在这里。

#### 懒标记的关系

- 这题有三类修改：赋 `0`、赋 `1`、翻转。
- 因此懒标记之间的关系就不能只靠一个标记糊过去，必须分清楚覆盖和异或的先后关系。
- 你代码里的处理是：
  - `tag = -1` 表示当前没有覆盖标记
  - `tag = 0/1` 表示这一段被整体赋成 `0/1`
  - `tag_xor` 表示这一段是否还挂着一个翻转操作
- 如果当前节点已经有覆盖标记，再来一次翻转，那就直接把 `tag` 从 `0` 变 `1` 或从 `1` 变 `0`。
- 如果当前没有覆盖标记，那就把 `tag_xor` 取反。
- 这个关系处理对了，整道题就不会乱。

#### 做法小结

- 这题表面上是线段树模板，实际上核心在于维护信息的设计。
- 一旦你意识到翻转会让 `0/1` 完全对称，因此应该把两套状态一起维护，那么后面就会非常顺。
- 所以这题最值得记的，不是某一行代码，而是这种“为了配合操作而设计节点状态”的思路。

### [P5522 [yLOI2019] 棠梨煎雪 - 洛谷](https://www.luogu.com.cn/problem/P5522)

- 这题和前一道完全不是一个风格。
- 如果说 `P2572` 是经典线段树状态维护题，那么 `P5522` 更像是“状态压缩 + 区间合并”的题。
- 关键不在线段树本身，而在于怎么把一个区间里每一位的状态压成一个好合并的表示。

#### 每一位到底有什么状态

- 对于一个位置来说，区间内某一列只有四种可能：
  - 全是 `0`
  - 全是 `1`
  - 全是 `?`
  - 同时出现了 `0` 和 `1`，也就是冲突
- 一旦这一列冲突，那么整个区间答案直接变成 `0`。
- 如果这一列全是 `?`，那这一位可以自由取 `0/1`，因此会给答案贡献一个额外的乘 `2`。
- 如果这一列被确定成 `0` 或 `1`，那这一位就没有额外贡献。

#### 为什么可以压状态

- 题目本身的列数不大，所以每一列的四种状态可以直接压进一个整数里。
- 常见做法是把每一位状态看成一个 `4` 进制位：
  - `0` 表示冲突
  - `1` 表示确定为 `0`
  - `2` 表示确定为 `1`
  - `3` 表示全是 `?`
- 这样一个字符串对应的整体状态就可以压成一个整数。

#### 区间合并为什么是按位与

- 这是这题最漂亮的地方。
- 因为我们关心的是“这一列在整个区间里还能不能自洽”，所以合并两个子区间时，本质上是在做限制的交。
- 例如：
  - 左边说这一位只能是 `0`
  - 右边说这一位只能是 `0` 或 `?`
  - 合并后仍然只能是 `0`
- 如果左边要求 `0`，右边要求 `1`，那就直接冲突。
- 所以很多写法里会把每一位状态编码成适合做按位与的形式，这样整个区间的信息合并就会非常快。
- 于是线段树上每个节点维护的其实不是普通数值，而是一整个“区间可行状态”。

#### 查询答案怎么还原

- 查询到一个区间后，我们拿到的是这个区间所有列压缩后的整体状态。
- 然后逐列拆开判断：
  - 如果某一列是冲突态，答案就是 `0`
  - 如果某一列是全问号态，答案乘 `2`
  - 其他情况不变
- 最后根据乘法原理，就能得到这一段的总方案数。

#### 这题和普通线段树题的区别

- 普通线段树题常见的是维护区间和、最值、最长连续段这些量。
- 这题不是维护一个“数值答案”，而是在维护“答案的约束状态”。
- 所以更准确地说，它考的是：
  - 如何把局部信息压缩成一个容易合并的表示
  - 如何让区间合并操作尽可能简单

#### 做法小结

- `P2572` 练的是“维护什么状态”。
- `P5522` 练的是“状态怎么压，怎么合并”。
- 这两题放在一起看，其实挺有代表性：线段树不一定总是维护区间和，它更像是一个维护“可合并区间信息”的通用框架。

## 考研

- `built-in`：自带的 / 内置的 / 嵌进去的
- `__builtin_popcount` 对应的理解就是：统计一个整数二进制表示里 `1` 的个数
- 在很多状压题里，`__builtin_popcount(x)` 经常就是直接拿来数状态里有多少位被选了
